This chapter explains the pipeline structure set up to validate the LLM summaries generated for the Cloudspeakers Newsletter. The pipeline consists of five modules, building the dataset, decomposing the summary into atomic claims, parsing the Dutch source text, validating the of the claims against the source text or external database knowledge from MusicBrainz and lastly evaluating the outcome. 

In order to correctly verify the summary given by the LLM model, we have to create a pipeline that sources the needed context. As seen in Chapter \ref{Newsletter}, the newsletter contains a one-sentence summary. Before we can validate whether the sentence is correct, we need to extract all context. This is done by loading the URL using a headless browser, where we scrape the Dutch text. This then gets parsed and cleaned, and sent to be validated by a NLI model. Meanwhile, the summary gets separated into atomic claims, to be validated. The NLI model consists of a Transformer-based language model which can perform cross-lingual comparisons, which compares the source, our premise, to the hypothesis, atomic claims. This pipeline follows the \textit{segmentation, matrix generation, aggregation} structure suggested by Laban \etal \cite{laban2021summaC}. However, in our experiment, we supply our baseline evaluation with the external knowledge graph, to aid the program in finding and correcting hallucinations. We also do not use any of the mean aggregation methods, but instead opt to use \textit{min aggregation}. Using entailment scores of our claims, we are able to classify two types of hallucinations.

\section{Corpus Construction} \label{rsh:datacorpus}
Before any validation or evaluation is possible, we need to load the source data given in the newsletters' \texttt{URL} field. This allows us to create a so-called \textit{corpus} which contains all available data we have on the artist we are evaluating. The corpus setup is implemented by loading all the data up front, so we save time loading all data again when our pipeline crashes later on, or if we wish to run the experiment again. This process collects the data in three steps: loading the local venue data, fetching the MusicBrainz facts, and saving everything to a JSON file.

\subsection{Extracting Venue Data}
As mentioned in Chapter \ref{Newsletter}, our primary data source is a generated newsletter, where we will be focussing on the venue Doornroosje, located in Nijmegen. This is set up using a script, where the primary event details and the LLM-generated summaries are manually extracted from these newsletters and provided to the script.

The events page gets opened using a chromium headless browser, set up using the \texttt{Playwright} package. This is done since we wanted to ensure that our web scraping was done in a dynamic and deterministic fashion and we could replace the venue if we wanted to. This meant the option of using an AI Webcrawler was out of the equation, due to it not being deterministic and it being to cost heavy to set up. To prevent memory leaks and segmentation faults during the batch processing, we allocated a single browser context and passed every \texttt{URL} field from the input newsletters iteratively. 

Once we collected the HTML using the headless browser, we used \texttt{bs4} from the \texttt{BeautifulSoup} package to find the \texttt{main},  \texttt{body} or  \texttt{article} which contained the text originally scraped by the LLM. Other HTML tags were discarded. This source text is then passed on to our Dutch Parser module, which parses, clean and prepares the Dutch text to be used as the premise of the NLI check. 

\section{External Knowledge Graph Implementation}
The external knowledge graph we chose for this experiment is the MusicBrainz Database \cite{musicbrainz}.
MusicBrainz is a community-maintained, open source encyclopedia of music information. It stores all of the music information that is collected by users, from artists and their releases to works and their composers. MusicBrainz gives us the availability to use this service by calling their API, allowing us to automatically request information. 


\subsection{MusicBrainz Data Extraction}
Once the primary event data is manually set up, the script automatically fills the dataset with verified real-world facts by querying the MusicBrainz API. To respect the strict rate-limiting policies enforced by the MusicBrainz infrastructure, our script pauses for a second, before querying the next artist.

The main task of this module is to clean the extracted text, since HTML source text delivered by the corpus module contains boilerplate text that can clutter up our analysis. One factor that we noticed, was a constant section of a cookies pop-up text. During test runs of the web scraping, our corpus was filled with the paragraph that can be found in Listing \ref{lst:cookiesparagraph}.

\noindent Instead of filtering this text statically, which in theory would work since we are only scraping from a single source, we opted for a more dynamic approach to ensure that when this pipeline is adapted to another webpage less work is needed. This dynamic approach is heuristic-based, which works since basing our rules on shallow text features for boilerplate removal achieves high accuracy without the need for complex structural features or site-specific engineering \cite{Kohlschütter2010boilerplatedetection}. Our code utilizes regular expressions (regex) to search and remove Listing \ref{lst:cookiesparagraph}. First, we define a list of patterns, which are the commonly found phrases in cookie banners, such as ``We gebruiken cookies'' and ``Functionele cookies''. Then, we match the rest of the sentence and loop through the document and replace any matching string with an empty string, deleting the snippet. Lastly, we clean up large chunks of white space. 

% Explain the Over: {artist} 
After having removed the boilerplate text from the source, we need to append context about the artist to each sentence. As the validation is done over every single source line, context can be lost due to the artist name not being in the sentence. To solve this issue, we prepend the following string: ``\texttt{Over \{artist\_name\}:}'', where the name of the artist is inserted. This addition ensures that the sentences keep their main focus, which is a necessity for the validation process in Section \ref{rsh:validator}. 
The clean, contextualised Dutch source sentences are then passed on to the validator. All these individual sentences form the source sentence set $\mathcal{S} = \{ s_1, s_2, \dots, s_n \}$. 


\section{Decomopsing Summaries Into Atomic Claims}\label{rsh:decomposer}
The main task of the decomposer module is to separate the claims made by the LLM in the generated summary. A summary can contain from one until multiple claims, so it is important that all of them get separated into \textit{atomic claims} and get evaluated. We define a claim as \textit{atomic} only contains one fact and all atomic claims are mutually exclusive and collectively exhaustive which allows the subsequent NLI model to render a definitive hallucination label without semantic noise. This falls back on the first step of Laban \etal's \cite{laban2021summaC} structure, which exists of \textit{segmentation}, where we decompose a summary $\mathcal{C}$ into atomic claims $\mathcal{C} = \{ c_1, c_2, \dots, c_m \}$. 


\subsection{Decomposing Intuition}
While state-of-the-art frameworks like Google DeepMind's SAFE \cite{safe2024longform} rely on LLM prompting to decompose long-form summaries into atomic claims, introducing yet another machine learning model and creating high operational and computational costs, we made the choice to do this in a deterministic fashion. Our methodology adopts the core multi-stage atomic decomposition paradigm pioneered by SAFE, but replaces the LLM parser with a deterministic, rule-based linguistic dependency parse utilizing \texttt{spaCy}. Splitting claims into atomic claims, has been proven to improve performance in NLI for multiple models \cite{huang2026atomicsnlifinegrainednaturallanguage,laban2021summaC}. Our decomposer has been developed in parallel to the framework given by Huang \etal \cite{huang2026atomicsnlifinegrainednaturallanguage}. 


By utilizing a deterministic linguistic dependency parser rather than a secondary LLM for claim decomposition, we avoid the `recursive hallucination' risk \cite{yoo2026torlite}. When an LLM is used to decompose a summary, it may introduce its own hallucinations during the parsing phase. Our rule-based approach ensures that the decomposition remains faithful to the text's syntactic structure, effectively providing a stable, verifiable input for the subsequent NLI evaluation.

\subsection{Decomposer Implementation}
The core of our decomposer consists of treating a sentence as a hierarchical Dependency Tree, where words are nodes connected by functional relationships. These relationships mimic the function of words in English sentences, such as a \textit{preposition}, \textit{subject}, \textit{object}. The enumeration below shows the implementation of the algorithm. 
\begin{enumerate}
    \item Convert the raw text into a machine-readable dependency tree. This allows the system to identify the grammatical roles of words, such as which terms function as the subject and which function as the object of an action.
    \item Iterate through every token in the document to locate ``action anchors." The system prioritizes VERB and AUX (auxiliary verbs such as `is' or `has'), as these words define the core events or states necessary for a factual claim.
    \item Resolving Components: 
    \begin{itemize}
        \item Subject: Identify the nominal subject (`nsubj'). If the subject is a pronoun (e.g., ``He"), replace it with the primary entity name to ensure the claim remains contextually independent and verifiable.
        \item Verb Phrase: Collect all auxiliary verbs to preserve the original tense (e.g., ``is performing").
        \item Object: Locate the grammatical receiver of the action (direct object, attribute, or complement).
        \item Modifiers: Identify surrounding phrases that provide extra context, such as time, location, or manner.
    \end{itemize}
    \item  Combine the subject, verb phrase, and object to construct a coherent, standalone sentence that represents the core fact.
    \item If the sentence includes modifiers, ``multiply" the base statement by each modifier to generate individual, independent claims. This step converts one complex sentence into multiple simple, verifiable statements.
    \item  Perform final data hygiene. Remove exact duplicate claims and discard any fragments that are under 5 words in length. This ensures the final output contains only high-quality, meaningful information.
\end{enumerate}
The atomic claims set has now been created and can be validated by the next module. 

\section{Validating Atomic Claims Based on Source Text}\label{rsh:validator}
Once the atomic claims have been generated and the source text has been cleaned and contextualised, we can start the validation process. The validation module of the pipeline compares the hypothesis, atomic claims, against the premises, contextualised source text, and returns three different scores, \texttt{entailment}, \texttt{contradiction} and \texttt{neutral}. Based on these scores, we can determine the validity of the claim. Since the source text differs from the atomic claims, we will be using a cross-language NLI (XNLI) model, to ensure no context is lost in a (seperate) translation layer.

\subsection{Validation Design Choices}
As mentioned in Section \ref{introduction}, the option of using a generative model for the entailment check was discarded to avoid the inherent bias of self-validation where a LLM acts as an uncritical judge of its own generated output. Since the research objective is to establish the viability of a fully deterministic pipeline, we selected Facebook's \rob \cite{conneau2020unsupervised}. \rob, according to Conneau \etal, serves as an extensively documented `gold standard' baseline in cross-lingual NLI literature. 

\rob is a Transformer-based language model trained on 2.5 Terabytes of filtered CommonCrawl data across 100 different languages.
\rob learns a shared mathematical representation of human language, which results in it being able to convert multiple languages into a mapped mathematical vector space, meaning different languages can be compared. 
Conneau \etal show that \rob is very competitive with strong monolingual models on XNLI benchmarks, demonstrating that it is possible to have a single large model for all languages. \cite{conneau2020unsupervised}.
For the Cloudspeakers Experiment (Section \ref{Newsletter}), this saves us steps from translating between Dutch and English which could lead to information and/or performance loss, or limiting the experiments to pages that \textit{do} have an English variant. 
As mentioned in \ref{pre:sec:nli}, NLI looks at the entailment, contradiction and neutral scores. \rob outputs these when given a premise and hypothesis and using these we can determine likelihood of a hallucination by the LLM. \subsection{\rob Implementation}
The \texttt{validator} is implemented in Python utilizing PyTorch and the HuggingFace \\ \texttt{transformers} library, using the specific \texttt{joeddav/xlm-roberta-large-xnli} model. 
\noindent As mentioned in Section \ref{rsh:decomposer}, the pipeline delivers multiple different atomic claims ($c^A_1, c^A_2, \dots, c^A_n$) for every summary $\mathcal{C}^A$ and the source sentences $S^A = \{s^A_1, s^A_2, \dots, s^A_m\}$ for every artist $A$. Since \rob is set up to only check one premise against a single hypothesis, we will run the model for every claim and every sentence in the source material and find which sentence results in the highest entailment and which in the highest contradiction. We chose to run the experiments in this way, to adhere the the segmentation framework \cite{laban2021summaC} and to remain below \rob's maximum input token capacity. To be certain we won't go above \rob's 512-token input limit, we tokenize the pairs ($c^A_i, s^A_j$) with truncation enabled. This is just a precaution, because during our experiments \rob never reached this limit. 

The raw numbers (logits) output by the model are passed through a Softmax function, which distributes values across all the classes accordingly, to calculate a probability distribution across the earlier mentioned classes: \texttt{contradiction}, \texttt{neutral} and \texttt{entailment}. This ensures they all add up to a value of $100\%$.

The script holds two trackers for each claim in $mathcal(C^A)$:
\begin{enumerate}
    \item $\texttt{ent\_score}_{\text{max}}(c^A_j)$: The maximum entailment score of Claim $j$ for all sentences $S^A$. 
    \item $\texttt{con\_score}_{\text{max}}(c^A_j)$: The maximum contradiction score of Claim $j$ for all sentences in $S^A$. 
\end{enumerate}

\noindent These scores and sentences are then passed to the evaluation module; together, the two trackers represent the peak values of the $M \times N$ pair matrix described by Laban et al \cite{laban2021summaC} with every possible pair $(c_i, s_j)$. However, we are not interested in the entire matrix, but just the peaks, so we do not store the entire matrix but just the max scores. This is due to the labelling logic will apply in Section \ref{rsh:tenrary}. 

\section{Evaluating Validation Scores}\label{rsh:eval}
The evaluation of the newsletter summaries, will be done in a two step process. After \rob has finished generating validation scores for each claim of a summary sentence, we will first label the claims as \texttt{SUPPORTED}, \texttt{INTRINSIC} or \texttt{EXTRINSIC}
%\cite On Faithfulness and Factuality in Abstractive Summarization", Maynez et al. (2020), \source{Add this one}
\cite{huang2025survey}
  based on the Dutch source text. Flagging the claims will be done based on the trackers set up in Section \ref{rsh:validator}. Next, after the hallucinations have been marked, we aggregate these results in based on the SAFE framework \cite{wei2024longform}, by using a strict atomic verification, which entails that every claim of the sentence has to be marked \texttt{SUPPORTED} for a sentence to be given the \texttt{CORRECT} verdict. Otherwise, it receives \texttt{INCORRECT}. 
  
  We choose this binary evaluation approach, since we use a SummaC-based \cite{laban2021summaC} baseline approach and most other NLI models \cite{wei2024longform} make use of a binary evaluation. 

\subsection{Ternary Atomic Claim Evaluation}\label{rsh:tenrary} 
\noindent To correctly map every atomic claim onto one of the three claim labels (\texttt{SUPPORTED}, \texttt{INTRINSIC} or \texttt{EXTRINSIC}), we make use of the trackers set up in Section \ref{rsh:validator}. Equation 4.1 shows when a label is applied. 

\begin{equation}
\text{Verdict}(c_j) = 
\begin{cases} 
\text{SUPPORTED} & \text{if } \texttt{ent\_score}_{\text{max}}(c_j) \ge \mathcal{T} \quad \text{and} \quad \texttt{ent\_score}_{\text{max}}(c_j) > \texttt{con\_score}_{\text{max}}(c_j) \\
\text{INTRINSIC} & \text{if } \texttt{con\_score}_{\text{max}}(c_j) \ge \mathcal{T} \quad \text{and} \quad \texttt{con\_score}_{\text{max}}(c_j) > \texttt{ent\_score}_{\text{max}}(c_j) \\
\text{EXTRINSIC} & \text{otherwise}
\end{cases}
\end{equation}
For all $c_j$ in $\mathcal{C}$, $\texttt{ent\_score}_{\text{max}}(c_j)$ is that claims highest entailment score for sentence $s_j$ , $\texttt{con\_score}_{\text{max}}(c_j)$ is that claims highest contradiction score for sentence $s_j$ and $\mathcal{T}$ referring to the confidence threshold. 



\subsection{Binary Sentence Evaluation}
After having given a verdict to the claims of a sentence, we start the aggregation process of these claims. The form of aggregation we have chosen is from the SAFE framework \cite{safe2024longform}, where every claim must hold before a parent sentence can hold. While the SummaC \cite{laban2021summaC} architecture utilizes mean-pooling to aggregate summaries,this approach risks missing hallucinations due to a sentence having a large number of claims. We opt to implement \textit{min-pooling}: 

\begin{itemize}
    \item \textbf{CORRECT:} A sentence is labeled as factually consistent if and only if all of its atomic claims are classified as \texttt{SUPPORTED}.
    \item \textbf{INCORRECT:} A sentence is labeled as inconsistent if at least one of its atomic claims is classified as an \texttt{INTRINSIC} or \texttt{EXTRINSIC} hallucination.
\end{itemize}
This strict binary output serves as the final evaluation metric for our pipeline, allowing us to calculate the error detection rate when comparing the baseline model against the experiment model, which uses the external knowledge graph.

\section{Experiment setup (Knowledge Graph Integration)}
To asses whether a hybrid evaluation framework, implementing \rob with the MusicBrainz integrated knowledge graph, can effectively detect and classify intrinsic and extrinsic hallucinations, we have set up our discussed framework. At the point of evaluation, we will see whether inserting the external data can improve our sentence verdict. By helping the validation module by inserting more data we confirmed holds, we hope to improve it's overall score. 

\subsection{Baseline Evaluation Protocol}
To compare the results, we will first gather results without the external database, to create a baseline. This baseline can be loosely compared to other NLI Summary Frameworks \cite{laban2021summaC}, as well as modern LLM-driven factuality evaluators \cite{chen2023felm}. For this baseline framework, we have to also calculate the optimal value for $\mathcal{T}$, mentioned in Equation 4.1. This is done after having generated the validation scores. The scores are saved, and we run a check for $ 0 \le \mathcal{T} \le 100$ in increments of $5$, creating 21 unique thresholds. We choose this size of increments, since smaller increments may lead to overfitting of the model. Once the optimal threshold for our baseline setup has been calculated, we apply this threshold to the experiment setup to guarantee that possible improvements can be attributed to our addition of an external knowledge graph. 

\subsection{External Knowledge Graph Override}

Since \rob is built on comparing natural language, we cannot compare the values from the MusicBrainz API directly to our hypothesis. For this, we will need to generate synthetic source text. For every field listed in the Table \ref{tab:mb_synthesis_templates} that contains data in the MusicBrainz database, we fit it into a custom template sentence. These sentences can be found in Table \ref{tab:mb_synthesis_templates}. Due most of the time, the \texttt{place} being ``Unkown'' and the \texttt{type} of entity not being relevant, we omit these.

\subsection{Manual Ground Truth Labelling}
To check the accuracy our labels for claims and sentences, we have manually marked every sentence and claim, so that the results can be compared. Creating ground-truth labels for the claims, which leads into the ground-truth for the sentences, was done based on the following considerations:
\begin{itemize}
    \item  \textbf{Completeness}: Every topic mentioned in the summary had to be able to be traced back to (a part of) a sentence.
    \item \textbf{Factuality}: All claims had to be truthful according to the source text, meaning no contradictions were found in the source.
\end{itemize} 
\noindent Notably, we based the information on the source, and not global knowledge. This entails that if we knew that e.g. ``Adult DVD'', wasn't from The Netherlands, because we might have been to one of their concerts, and the summary claimed that they were, we mark the claim as \texttt{EXTRINSIC} since the knowledge of where ``Adult DVD'' originates from is not present in the source material. If we were to update the source with knowledge of the ``Adult DVD's'' origin, or extend it using another database, the hallucination will be labelled as \texttt{INTRINSIC}. These ground truth labels can be found in Appendix \ref{tab:ground_truth_labels}.  

\subsection{Evaluation Metrics}
To evaluate the performance of both the baseline and experimental pipelines, we use the following metrics: 
\begin{itemize}
    \item \textbf{Threshold Optimization:} To assign the confidence threshold ($\mathcal{T}$) we derive it by evaluating the continuous entailment scores of the baseline model against the ground truth and locking the value that maximizes the \textbf{F1-score}.
    
    \item \textbf{Atomic Claim Evaluation:} At the component level, we utilize a standard confusion matrix, alongside \textbf{Precision, Recall, and F1-scores}, to measure the cross-lingual NLI model's ability to accurately categorize the ternary labels (\texttt{SUPPORTED}, \texttt{INTRINSIC}, \texttt{EXTRINSIC}). This level of measurement is critical for quantifying exactly how many claims are impacted by the external knowledge graph.
    
    \item \textbf{Binary Sentence Evaluation:} At the final aggregation level, where the min-aggregation assigns a definitive \texttt{CORRECT (1)} or \texttt{INCORRECT (0)} label, we evaluate the overall system using standard binary classification metrics: \textbf{Accuracy, Precision, Recall, and F1-score}. 
    
     To view the impact of the external knowledge graph integration, we emphasize two targeted metrics within this classification:
    \begin{itemize}
        \item \textbf{Error Detection Rate (Recall):} The proportion of ground-truth hallucinations successfully flagged by the pipeline.
        \item \textbf{False Positive Rate:} How often the pipeline falsely accuses a summary of containing an error. 
    \end{itemize}
\end{itemize}
